Sort Patterns for Authoring Interfaces

Research into how products handle sorting in views where users are also creating and editing data. Filtering is a separate exploration — this focuses purely on sort behavior.

The Core Problem

When a timesheet has an active sort, entries move around without the user doing anything — editing an hour value causes the row to teleport, and newly created entries land in unexpected positions. The user's spatial model of "where I put things" is broken.

This matters most for timesheets because users are constantly switching between authoring (adding/editing entries) and reviewing (scanning for errors, checking totals) in the same view.

Jobs to Be Done

Why would someone sort a timesheet? The answer shapes the solution.

JobWhat They Actually Need
Find entries that need fixingSurface broken entries — may not need sort at all (see "Needs Attention" exploration, documented separately)
Check if hours look rightMomentary reorder by duration — then go back to editing
Review by statusSee what's released vs pending — a temporary lens, not a permanent arrangement
Organize my wayManual drag-drop to arrange entries in the order that makes sense to me

Most of these are momentary — the user wants to see a different arrangement briefly, then return to their own order. This is key to choosing between the approaches below.

How Other Products Handle This

Google Sheets / Excel

One-Shot Sort

Sorting is a destructive one-time action. Rows rearrange, then the view returns to manual order. No "live sort." Original order is lost unless you had a helper column.

Airtable

"Keep Sorted" Toggle

"Automatically sort records" toggle. ON: live-sort, drag disabled, sorted columns get orange highlight. OFF: sort applied once, manual reorder available.

Linear / Todoist

Manual is Default

"Manual" is a first-class sort option. When any other sort is active, drag-to-reorder is disabled. Clear either/or: your order or the system's.

Most SaaS tables

Always-Active Sort

Click a column header to sort. Sort persists. Entries re-sort on every change. No manual ordering. This is what Laurel does today.

Tradeoffs

Strengths Simple to understand: "it's sorted by X." No ambiguity about what the order means. Familiar from most data table UIs. Good for pure review/read tasks.
Weaknesses Entries teleport when edited — spatial instability during authoring. No manual drag-drop ordering while sort is active. Users can't maintain their own preferred order. New entries land wherever the sort puts them, which can be disorienting.

One-Shot Sort

Sort is an action, not a mode. Click a sort option → entries rearrange once → you're immediately back to manual ordering. Like shuffling a deck of cards into a new arrangement. Drag-drop is always available.

Tradeoffs

Strengths Spatial stability after sort — entries stay where they land. Drag-drop always available. Supports the "momentary reorder" JTBD well. No persistent state to manage or communicate. Sort button stays inactive — no mode to get stuck in.
Weaknesses Different from what users expect if they come from always-active-sort tools. No way to maintain a persistent sort. Original order is gone after sorting (unless you manually re-drag). Power users who want "always sorted by status" can't get that here.

Auto-Sort Toggle

Sort rules can be set, and a "Keep sorted" toggle controls whether sorting is live (entries re-sort on edit, drag disabled) or one-shot (rearrange once, then freely reorder). Gives users explicit control over the tradeoff.
